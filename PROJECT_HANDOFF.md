# 종로 도보 동선 추천 (가칭: Jongno Walk-Course Recommender) — 프로젝트 핸드오프 문서 **v2**

> **이 문서의 목적**
> 채팅이 길어지거나 이미지 수 제한으로 새 창에서 다시 시작해야 할 때, **이 문서 전체를 새 채팅에 붙여넣으면 맥락이 그대로 복원**되도록 만든 단일 진실 소스(single source of truth)입니다.
> 새 채팅 첫 메시지 예시: *"아래는 진행 중인 학교 과제의 핸드오프 문서야. 이걸 기준으로 이어가자. 현재 상태는 13번 섹션을 봐줘."* + (이 문서 붙여넣기)
>
> **v2 변경 요지:** NN을 "랭커 부속"에서 **프로젝트 주인공**으로 격상(직접 구축·학습하는 NN 사다리 + 베이스라인 비교). 그간 확정 사항 반영(React+Leaflet/GitHub Pages, RL은 v2, Claude Code는 Max 플랜 포함, 레포 스캐폴딩·git 초기화 완료).

## 1. 프로젝트 한 줄 정의
**종로구 지도 위에서, 내 출발점으로부터 도보 예산(30분/1h/2h/3h) 안에 닿는 스팟(맛집·관광지·핫스팟)을 우선순위로 추천하고, 하나를 고르면 다음 후보가 다시 떠서 A→B→C 동선이 완성되는 무료 웹앱.**

- 핵심 상호작용: `A 선택 → B-1…B-n 우선순위 표시 → B-11 선택 → C-1…C-n …` (예산 소진까지)
- **학습 목표(중요):** NN을 *직접 짜고 학습시키는* 경험이 이 과제의 중심. 추천 = NN, 시간/도달성 = 결정론.
- 학술 프레이밍(선택): 보행 접근성(walkability) · 시간지리학의 time-space prism

## 2. 절대 제약 (Hard Constraints)
1. **전부 무료.** 유료 API·유료 빌링 금지. 무료 등록(키 발급)은 허용. (Claude Code는 사용자의 **Claude Max 플랜에 포함**되어 추가비용 없음 → 제약 위반 아님.)
2. **배움 중심.** 결과물보다 학습 경험·재현 가능성 우선. 특히 **NN을 적극적으로 구축·학습**.
3. **지도에 경로와 추천이 반드시 보여야 함.** (텍스트 리스트만으로는 불가)
4. **경계 침범 허용·안내.** 도보 예산이 크면 종로구를 벗어나 **중구·서대문구·성북구·은평구**(및 경계상 동대문구·강북구 일부)로 넘어갈 수 있음 → 데이터·네트워크가 이 버퍼를 포함해야 하고, UI는 다른 구로 넘어감을 표시.
5. 식당 폐점 여부 검증은 **사용자가 직접 처리**(앱 범위 밖).

## 4. 아키텍처 (4 레이어)
```
(A) 데이터 레이어   : 종로+버퍼 POI 테이블(피처) + 도보 OD행렬   [오프라인 사전계산]
(B) 도달성 엔진     : OD[현재][후보]+체류[후보] ≤ 남은예산 인 후보만 통과  [결정론적, NN 아님]
(C) 점수·추천 레이어 : 통과 후보 정렬  ← ★ NN이 사는 곳 (이 프로젝트의 두뇌)
        점수 = 학습랭커(궤적,후보) ⊕ 사용자가중신호 ⊕ 성·연령 인기도조건화
               ↑ NN(직접 구축·학습)   ↑ 언급량·군집·생활인구(슬라이더)  ↑ 집계
(D) 화면(웹)        : 지도 + 컨트롤(예산·신호 슬라이더·인구 필터) + 우선순위 마커 + 동선 폴리라인
                      (+ 보너스: 후보 점수를 노드 크기·색으로 = "층층이 켜지는 NN" 시각화)
```
**원칙:** 하드 제약(시간 예산)은 (B)가 정확히 거르고, 그 안에서 (C)의 NN이 순위만 매긴다.

## 5. ★ NN 사다리 (이 과제의 핵심 — 직접 구축·학습)
> **방침:** Foursquare 등 공개 체크인으로 *방법을 학습·검증*(정량지표), 그 구조를 종로 POI에 적용(데모). 각 레벨에서 **모델 정의 → 학습 루프 → loss 관찰 → 평가**를 직접 손으로.

| Level | 모델 | 직접 경험하는 것 | 산출/근거 |
|---|---|---|---|
| **L1** | MLP 랭커 (`nn.Linear`+ReLU → 점수) | 모델 정의·순전파·손실·옵티마이저·학습 루프 | NN 입문 완주, Recall@k/NDCG |
| **L2** | 학습된 임베딩 (`nn.Embedding`) | 표현 학습, UMAP로 임베딩 시각화 | POI가 종류별로 뭉치는 그림 |
| **L3** | 시퀀스 모델 (LSTM / 소형 Transformer) | A→B→C를 *순서*로 인코딩, 다음 장소 예측 | "층층이 펼쳐지는 NN" 실구현 |
| **L4** *(선택)* | GNN (POI 그래프 메시지 패싱) | 그래프 표현학습 | 고급 확장 |
| **v2** *(연기)* | 강화학습 정책 (MDP) | 전체 동선 보상 최적화 | future work |

**베이스라인 비교(과학적 서사):** 인기순·KNN·로지스틱·결정트리·SVM·앙상블을 비교군으로 두고 **위 NN과 Recall@k·NDCG로 대결** → "고전 대비 NN 우위"를 수치로 증명. (= NN이 장식이 아니라 *증명된 선택*)

**표현/전처리 기법(보조):** DBSCAN(핫스팟 군집), PCA/UMAP(차원축소·시각화), 판별분석/선형회귀(보조 분류·예측).

## 7. 데이터 수집 범위 & 파이프라인
- **공간 범위:** 종로구 경계 + 약 2~4km 버퍼(3시간 도보 ≈ 13km). 버퍼가 중·서대문·성북·은평 포함.
- **노드 캡:** v1은 ~500~800개로 제한(카테고리별 기초 relevance 상위)해 OD행렬을 다룰 만하게.
- **파이프라인:** bbox/구경계 → POI 수집(중복제거) → 피처 보강(언급량·생활인구·핫스팟 점수) → 테이블(parquet) → OSMnx 보행망 → 노드 간 OD행렬 사전계산 → 저장.
- 이 수집 파이프라인 자체가 GitHub 산출물.

### 핫스팟 점수 (전처리, NN 아님)
```
hotspot_score = z(블로그/검색 언급량) + DBSCAN 군집 밀집도  → 0~1 정규화 → 노드 피처
```
